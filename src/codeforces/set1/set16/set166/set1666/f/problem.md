# Problem

Little Fiona has a collection of `n` blocks of various sizes `a1, a2, ..., an`, where `n` is even. Some blocks can have equal sizes. She wants to use all blocks to build a stack.

Let `b1, b2, ..., bn` be block sizes in the stack from top to bottom. Since all blocks are used, `b` must be a permutation of `a`.

Fiona calls a stack **fancy** if both conditions hold:

1. Alternating inequalities:
   `b1 < b2 > b3 < b4 > ... > b(n-1) < bn`
2. Even-position values are strictly increasing:
   `b2 < b4 < b6 < ... < bn`

Two stacks are different if they differ at least in one position.

Find the number of different fancy stacks modulo `998244353`.

## Input

The first line contains integer `t` (`1 <= t <= 2500`) — the number of test cases.

Each test case consists of:

- an integer `n` (`2 <= n <= 5000`, `n` is even),
- a non-decreasing array `a1, a2, ..., an` (`1 <= a1 <= a2 <= ... <= an <= n`).

It is guaranteed that the sum of `n` over all test cases does not exceed `5000`.

## Output

For each test case, print the number of fancy stacks modulo `998244353`.

## Example

**Input**

```text
2
4
1 2 3 4
8
1 1 2 3 4 4 6 7
```

**Output**

```text
2
4
```

### solution

这题的关键不是直接构造排列，而是把条件重新解释成：

- 偶数位形成一个严格递增序列；
- 每个奇数位都必须严格小于它右边那个偶数位；
- 中间的奇数位还必须严格小于它左边那个偶数位。

因为偶数位本身严格递增，所以如果我们从 **大到小** 处理值，那么“某个值能不能放到偶数位”只和当前已经放了多少个更大的偶数位、以及目前还有多少空的奇数位可以接收更小元素有关。

---

#### 1. 先按值分组

原数组已经有序。把相同的值压成若干组：

- `cnt[1], cnt[2], ..., cnt[m]`

这里按值从小到大编号，所以后面 DP 时我们会从 `m` 往 `1` 处理，也就是从大值往小值放。

再记：

- `S[i] = cnt[i] + cnt[i+1] + ... + cnt[m]`

表示从第 `i` 组到最后总共有多少个元素。

---

#### 2. 为什么要从大到小处理

设最终偶数位有 `n/2` 个。

如果已经把所有 **比当前值大的** 元素都放好了，并且其中有 `j` 个被放到了偶数位，那么：

- 这 `j` 个偶数位的位置关系已经确定；
- 它们之间和两侧会形成一些“可以塞更小元素”的奇数位空槽。

对于当前更小的值，我们只需要知道：

- 现在已经有多少个偶数位被占了：`j`
- 还有多少个奇数位空槽能放当前值：这个量可以由 `j` 和已经处理过的元素数量推出

这就是为什么状态里只需要记 `j`。

---

#### 3. 可用奇数槽数 `get(j)`

如果已经放了 `j` 个偶数位上的元素，那么这些偶数位之间最多能提供多少个合法奇数槽？

- 如果 `j < n/2`，那么最左边那个偶数位左侧还不能立即放，因为它还缺少更小结构的完整性，所以只有 `j-1` 个内部槽有意义；
- 如果 `j = n/2`，说明所有偶数位都放完了，那么最终合法的奇数位正好有 `j` 个。

所以代码里定义：

```text
get(j) =
    j,             if j = n/2
    max(0, j-1),   otherwise
```

这表示：当已经固定了 `j` 个偶数位之后，理论上最多能容纳多少个“更小元素”的奇数位置。

---

#### 4. DP 状态

定义：

- `dp[i][j]`

表示：

- 已经处理完了第 `i..m` 组值（也就是所有更大的值都已经安排好了）；
- 其中恰好有 `j` 个值被选作偶数位上的元素；
- 当前合法方案数是多少。

初始状态：

- `dp[m+1][0] = 1`

因为一开始什么都还没放。

目标状态：

- `dp[1][n/2]`

即所有值都处理完，并且恰好选了 `n/2` 个偶数位元素。

---

#### 5. 当前组可以怎么放

设当前处理的是第 `i-1` 组，它的大小出现了 `c = cnt[i-1]` 次。

又设当前状态是 `dp[i][j]`。

这时已经处理过的元素总数是 `S[i]`，其中有 `j` 个被放到了偶数位，所以：

- 已经占掉的奇数位元素数是 `S[i] - j`

而目前理论上总共可用的奇数槽数是 `get(j)`。

所以真正还能给当前这一组使用的奇数槽数是：

```text
nowCan = get(j) - (S[i] - j)
```

这就是当前组能塞入的“剩余空槽”。

如果 `nowCan < 0`，自然无解；代码里通过排列数函数自动变成 `0`。

---

#### 6. 两种转移

当前这一组大小相同的 `c` 个元素，有两种选择：

##### 情况 A：这一组没有元素去偶数位

那么这 `c` 个元素都只能去当前可用的奇数槽里。

从 `nowCan` 个槽里依次放入 `c` 个元素，方案数是排列数：

```text
P(nowCan, c) = nowCan! / (nowCan - c)!
```

因此：

```text
dp[i-1][j] += dp[i][j] * P(nowCan, c)
```

---

##### 情况 B：这一组恰好拿出一个元素放到新的偶数位

注意：

- 同一个值不可能在偶数位出现两次，因为偶数位必须严格递增。

所以这一组如果进入偶数位，只能选 **1 个**。

选哪个副本，有 `c` 种方式。

剩下 `c-1` 个副本必须放到奇数槽里，因此还需要从 `nowCan` 个槽里安排 `c-1` 个：

```text
P(nowCan, c-1)
```

所以：

```text
dp[i-1][j+1] += dp[i][j] * c * P(nowCan, c-1)
```

---

#### 7. 为什么最后还要除以 `cnt!`

上面的 DP 在转移时，把同值元素当作“可区分副本”来计数了：

- 例如一组有 `c` 个相同数字，选哪个去偶数位记成了 `c` 种；
- 奇数位里对这些相同值的排列也都被重复算了。

但题目里相同大小的块是不可区分的，最终只看数值序列。

所以最后要对每一组除以：

```text
cnt[i]!
```

总共除以：

```text
∏ cnt[i]!
```

在模数下就是乘上逆元：

```text
ans = dp[1][n/2] * ∏ invFact[cnt[i]]
```

---

#### 8. 为什么这个 DP 是正确的

从大到小处理时，有两个非常重要的性质：

1. 当前值如果放到偶数位，那么它对应的是“当前最小的已选偶数位值”，不会和更大的偶数位冲突；
2. 当前值如果不放到偶数位，那么它只能填到已经由更大偶数位形成的合法奇数槽里。

因此，对于当前组来说，未来更小的值不会改变“它现在能不能放”的判定，只会去填剩下的槽。

也就是说，当前状态只需要知道：

- 已经选了多少个偶数位元素 `j`
- 由此导出的空槽数量 `nowCan`

就足够决定转移，不需要记更复杂的结构。

---

#### 9. 复杂度

设不同值的组数为 `m`，显然 `m <= n`。

DP 状态数：

- `O(m * n)`

每个状态转移是 `O(1)`。

所以总复杂度：

- `O(n^2)`

空间复杂度：

- `O(n^2)`

在 `sum(n) <= 5000` 的限制下完全可行。

---

#### 10. 为什么从大到小处理，而不是从小到大

这题最容易卡住的地方就在这里。

表面上看，从小到大、从大到小都像是可行的 DP 顺序，但实际上：

- 从大到小很好做；
- 从小到大则会让状态变得很别扭。

原因在于“当前元素是否合法”依赖的是 **过去** 还是 **未来**。

##### 从大到小：合法性只依赖已经放好的更大元素

当我们从大到小处理时：

- 更大的值已经决定了哪些元素被选进偶数位；
- 这些偶数位之间形成了若干合法奇数槽；
- 当前这个更小的值要么：
  - 作为一个新的偶数位元素，
  - 要么去填这些已经存在的奇数槽。

关键在于：

- 当前值能不能放，完全由“已经选了多少个更大的偶数位”决定；
- 未来更小的值不会反过来影响当前值是否可放。

所以状态只要记：

- 当前已经选了多少个偶数位 `j`

就够了。

这时问题是封闭的，转移非常干净。

##### 从小到大：合法性依赖未来

如果改成从小到大处理，就会遇到一个本质麻烦：

- 当前这个小值如果想放在奇数位，
- 它必须小于右边那个偶数位；
- 对于中间奇数位，它还必须小于左边那个偶数位。

但这些更大的偶数位，在“从小到大”的过程中往往 **还没有出现**。

也就是说，你现在并不能立刻判断：

- “把这个值放进某个奇数位，最后是否合法”

因为这取决于未来会不会有合适的更大值来把它“夹住”。

于是你就不得不记录很多“待匹配信息”：

- 现在有多少位置还在等未来的更大偶数位；
- 这些等待位置分别处于什么结构里；
- 它们以后能接受哪些值。

这会使状态明显变复杂，不再是一个简单的计数就能描述。

##### 一个直观比喻

可以把这题理解成：

- 从大到小：先搭骨架，再往里面塞小块；
- 从小到大：先塞小块，但你还不知道未来骨架能不能把这些小块合法包住。

前者的信息是已经确定的；
后者的信息是悬空的。

这就是为什么标准做法都选择：

- **从大到小处理值**

因为这样每一步的合法性都只依赖已经处理过的部分，DP 状态才能压得住。
